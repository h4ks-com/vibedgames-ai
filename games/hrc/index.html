HTTP RELAY CHAT

HRC Protocol Specification v0.1
IRC over plain HTTP. No WebSocket. No persistent connections.
Just POST requests bouncing through nodes like a game of hot potato.

💡 Core Concept

THE INSIGHT

Every chat protocol tries to maintain a persistent connection — TCP sockets, WebSockets, long-polling. This creates complexity: reconnection logic, heartbeat management, stateful servers, firewall issues.


HRC flips this entirely.

  • Every message is a standalone POST /relay request
  • Servers are completely stateless between requests
  • Messages hop through relay nodes via HTTP redirects/chaining
  • Clients don't need to hold any connection open
  • Works through corporate proxies, CGNAT, hotel WiFi — anything that lets you browse the web

Figure 3 — Message & Identity Data Model
  ════════════════════════════════════════════════
   MESSAGE ENVELOPE (what goes over the wire)
  ════════════════════════════════════════════════
  
  {
    "v":           1,              // Protocol version
    "type":       "message",      // "message" | "presence" | "system"
    "id":         "msg_a1b2c3",   // UUID, generated by sender
    "from":       "ed25519:abc...", // Sender's public key hash
    "to":         "#lobby",       // Destination: channel or "@user"
    "body":       "hello world",  // Encrypted payload (see below)
    "timestamp":  1709012345,     // Unix epoch seconds
    "parent":     null,           // For threading: parent msg ID
    "pow":        "0000a1b2",    // Proof-of-work nonce
    "sig":        "sig_hex...",  // Ed25519 signature over (id+to+body+ts)
    "hop_path":   ["relay-a","relay-b"], // Nodes this message passed through
    "ttl":        7               // Time-to-live in hops (default 7)
  }

Figure 4 — Complete Protocol Exchange (with E2EE)
  ═════════════════════════════════════════════════════
   SCENARIO: Alice joins #lobby and sends her first message
  ═════════════════════════════════════════════════════
  
  STEP 1 — DISCOVER A RELAY NODE
  ───────────────────────────────
  
  Alice:  GET https://relay.h4ks.com/hrc/status
          
  Server: {
            "version": "0.1",
            "node_type": "edge",
            "public_key": "ed25519:relay1...",
            "connected_peers": ["relay2.h4ks.com", ...],
            "channels": ["#lobby", "#random"],
            "max_payload": "64KB",
            "pow_bits": 16
          }
  
  STEP 2 — JOIN A CHANNEL (SUBSCRIBE)
  ───────────────────────────────────
  
  Alice:  POST https://relay.h4ks.com/hrc/join
          {
            "channel": "#lobby",
            "client_key": "ed25519:alice_pub...",
            "sig": "signature_of_channel_name"
          }
          
  Server: 202 Accepted
          {"subscribed": true, "since_id": "msg_fff..."}
  
  STEP 3 — SEND A MESSAGE (E2EE)
  ──────────────────────────────
  
  Alice's client does:
  
  1. Generate message UUID: msg_001
  2. Compose plaintext: {"text":"hey everyone!"}
  3. Encrypt for channel #lobby:
     → X25519 shared secret derived from:
       alice_privkey × each_member_pubkey
     → AES-256-GCM encrypt
     → ciphertext = "enc:v1:a1b2c3..."
  4. Compute PoW (hashcash, 16 bits):
     → find nonce where SHA256(msg+nonce) starts with 0000
  5. Sign everything:
     sig = Ed25519_Sign(alice_privkey, msg_id + to + body + ts)
  6. POST:
  
  Alice:  POST https://relay.h4ks.com/hrc/send
          {
            "v": 1,
            "type": "message",
            "id": "msg_001",
            "from": "ed25519:alice_pub...",
            "to": "#lobby",
            "body": "enc:v1:a1b2c3d4e5...",
            "timestamp": 1709012345,
            "pow": "nonce_3f4g",
            "sig": "sig_hex_..."
          }
          
  Server: 202 Accepted
          {"relayed": true, "path": ["relay1","relay2"]}
  
  STEP 4 — RECEIVE MESSAGES (POLL)
  ────────────────────────────────
  
  Bob:  GET https://relay.h4ks.com/hrc/poll?channel=%23lobby&since=msg_fff
       
  Server: 200 OK
          {
            "messages": [
              {
                "id": "msg_001",
                "from": "ed25519:alice_pub...",
                "to": "#lobby",
                "body": "enc:v1:a1b2c3d4e5...",
                "timestamp": 1709012345,
                "sig": "sig_hex_..."
              }
          ]
          
  Bob's client:
  1. Verify signature against alice's public key ✅
  2. Verify PoW ✅
  3. Decrypt body using shared secret → "hey everyone!"
  4. Display:  hey everyone!
  
  ═════════════════════════════════════════════════════
   THAT'S IT. 4 HTTP requests. Zero persistent connections.

🛡️ Anti-Abuse Design

Built-In Spam Resistance

  • Proof-of-Work per message — Client must find a 16-bit hashcash prefix before sending. Costs ~5ms on desktop, ~500ms on phone. Free to receive.
  • Rate limiting per Ed25519 key — Servers track send rate per identity, not per IP (defeats VPN rotation)
  • TTL-based loop prevention — Messages die after N hops. Can't amplify.
  • Signature verification required — Unsigned messages dropped at edge. No spoofing.
  • Optional: reputation scoring — Nodes can share bloom filters of known-spammer keys
