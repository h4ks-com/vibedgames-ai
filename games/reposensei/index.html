Lesson 1

What Even IS a Repository?

~5 min

💡 The Big Picture

A repository (or "repo") is just a folder that remembers everything. Every change you ever make to any file inside it — saved forever, with timestamps and your name on it. Think of it as a time machine for your code.

A repo contains three things:

📁 Your Files
Code, images, docs, configs — whatever you're working on
📜 History (Commits)
Every save point with who changed what and why
⚙️ Config (.git)
Hidden folder that holds all the magic — never touch it directly!
💡 Real world analogy: A repo is like Google Docs version history but for your entire project folder, works offline, and you control everything.

Lesson 2

Create Your First Repo

~8 min

💡 Two Ways To Start

You can either clone someone else's repo (copy it), or init a brand new one from scratch. We'll do both.

🆕 Method A: Start Fresh (git init)

Terminal — ~/projects
# 1. Make a folder for your project
$ mkdir awesome-project
$ cd awesome-project

# 2. Turn it into a git repo
$ git init
Initialized empty Git repository in /home/user/projects/awesome-project/.git/

# 3. Check status — should show no commits yet
$ git status
On branch main
No commits yet
nothing to commit (create/copy files and use "git add" to track)

📥 Method B: Clone Existing (git clone)

Terminal — ~/projects
# Clone someone else's repo from GitHub/GitLab/etc
$ git clone https://github.com/someone/cool-repo.git
Cloning into 'cool-repo'...
remote: Enumerating objects: 42, done.
remote: Counting objects: 100% (42/42), done.
remote: Compressing objects: 100% (30/30), done.
Receiving objects: 100% (42/42), 15.00 KiB | 1.23 MiB/s, done.
⚠️ Never run git init inside an existing repo! You'll create nested git repos which causes endless confusion. Always check if a .git folder already exists first.

Lesson 3

The Art of Committing

~10 min

💡 The Three-Step Dance

Every commit follows the same rhythm: edit → stage → commit. Think of it like packing a suitcase before a trip — you pick what goes in the bag (stage), then you zip it up and label it (commit).

Terminal — ~/awesome-project
# Step 1: Edit or create a file
$ echo "# Hello World" > README.md

# Step 2: See what changed
$ git status
Untracked files:
  (use "git add <file>..." to include)
	README.md

# Step 3: Stage the file (add to next commit)
$ git add README.md

# Step 4: Commit with a message explaining WHAT and WHY
$ git commit -m "Add README with project description"
[main (root-commit) a1b2c3d] Add README with project description
 1 file changed, 1 insertion(+)
 create mode 100644 README.md

# Step 5: View your commit log
$ git log --oneline
a1b2c3d (HEAD -> main) Add README with project description
💡 Pro tip: Write commit messages in imperative mood: "Add feature" not "Added feature". Imagine completing the sentence "If applied, this commit will..."

📋 Essential Commit Commands

git add file.txtStage one file
git add .Stage ALL changes
git add -pStage interactively
git commit -m "msg"Commit staged changes
git log --onelineCompact history view
git diffSee unstaged diffs
git diff --stagedSee staged diffs
git restore file.txtDiscard unstaged edits

Lesson 4

Branches & Pushing to GitHub

~12 min

💡 Branches Are Just Pointers

A branch is a movable label pointing at a commit. That's it. No magic. main points here, feature/login points there. You can move between them instantly. They cost almost nothing to create.

🌿 Branch Workflow

Terminal — ~/awesome-project
# Create a new branch and switch to it
$ git checkout -b feature/new-stuff
Switched to a new branch 'feature/new-stuff'

# Make some changes and commit
$ echo "new feature" >> stuff.txt
$ git add stuff.txt
$ git commit -m "Add new stuff feature"
[feature/new-stuff e5f6g7h] Add new stuff feature

# Switch back to main — your changes are safe on the other branch
$ git checkout main
Switched to branch 'main'

# Merge your feature branch INTO main
$ git merge feature/new-stuff
Updating a1b2c3d..e5f6g7h
Fast-forward | stuff.txt | 1 +
 1 file changed, 1 insertion(+)

☁️ Pushing to GitHub (Remote)

Terminal — ~/awesome-project
# Link your local repo to a remote (GitHub)
$ git remote add origin https://github.com/YOU/awesome-project.git

# Push your main branch to GitHub
$ git push -u origin main
Enumerating objects: 9, done.
Counting objects: 100% (9/9), done.
Delta compression using up to 8 threads
 * [new branch]      main -> main

# Next time, just:
$ git push
Everything up-to-date

# Pull changes FROM GitHub (when working with others)
$ git pull
Already up to date.
⚠️ Always pull before you push when collaborating! If someone else pushed while you were working, git pull merges their changes first. Skip this and you'll get rejected.

📋 Branch & Remote Cheat Sheet

git branchList local branches
git branch -aList ALL branches
git checkout -b nameCreate + switch branch
git merge nameMerge branch into current
git remote -vShow remotes
git push origin mainPush to GitHub
git pullPull + merge remote
git fetchDownload remote info

Lesson 5

Pro Moves & Survival Guide

~10 min

💡 The Commands That Save Lives

These are the commands you reach for when things go wrong. And things will go wrong. Memorize these or bookmark this page — they've rescued every developer at least once.

🚨 Oh Sh*t Commands

Terminal — Emergency Room
=== UNDO LAST COMMIT (keep changes) ===
$ git reset --soft HEAD~1
Changes staged. Commit message gone. Files intact. 🟢 SAFE

=== UNDO LAST COMMIT (discard changes too) ===
$ git reset --hard HEAD~1
⚠️ Changes GONE. Like delete-key gone. Use with fear.

=== DISCARD ALL UNCOMMITTED CHANGES ===
$ git checkout -- .
# OR (newer git):
$ git restore .
⚠️ All uncommitted edits = vaporized

=== SEE WHAT YOU CHANGED BEFORE STAGING ===
$ git diff

=== SEE A SPECIFIC FILE'S HISTORY ===
$ git log -p --follow -- main.py

=== GO BACK IN TIME TO OLD COMMIT ===
$ git checkout a1b2c3d
You're now in "detached HEAD" state. Scary name, harmless state.
Type 'git checkout main' to return to present.

✨ Quality of Life

Terminal — Comfort Zone
=== PRETTY LOG GRAPH ===
$ git log --oneline --graph --all --decorate
* e5f6g7h (feature/x) Add new stuff
* | a1b2c3d (HEAD -> main) Add README
|/
* 9i8j7k6 Initial commit

=== BLAME: WHO WROTE THIS LINE? ===
$ git blame main.py
^9i8j7k6 (Author 2024-01-15 10:23) import os
 a1b2c3d (You     2024-01-16 14:02) def main():

=== STASH: SAVE WORK FOR LATER ===
$ git stash
Saved working directory: "WIP on main: a1b2c3d"
# ... switch branches, fix bug, come back ...
$ git stash pop
On branch main: Changed file restored

=== CREATE .GITIGNORE ===
$ echo '__pycache__/
*.pyc
.env
node_modules/
.DS_Store
dist/
build/' > .gitignore
💡 The golden rule: Never force push (git push --force) to a shared branch. It rewrites history for everyone else and causes pain, tears, and angry Slack messages.
